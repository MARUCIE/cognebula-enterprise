Architecture Decision Record

灵阙融合多表面架构设计

4 路蜂群共识：从 4 个独立表面收敛为 2 个部署应用 + 2 个渲染变体。 架构边界由数据权限划定，不由 UI 复杂度划定。

v1.0 | 2026-03-27 | Swarm Consensus (Drucker / Jobs / Hickey / Hara)
Section 01

问题重述：为什么 4 个表面是伪命题

最初的提案包含 4 个独立用户界面：

#表面目标用户当前状态
S1PC 工作台客户（老板/CFO）已建成 10 页
S2专家工作台内部专家遗留 D3.js
S3运营后台平台运营者未建
S4移动客户端客户（手机）未建
蜂群判决：4 个表面是一种 Complection（编织）

真实结构是 2 种信息需求（客户 vs 内部）+ 2 种渲染密度（桌面 vs 移动）。 S2 和 S3 服务同一类用户（内部团队），深度不同而已。 S4 是 S1 的不同渲染方式，不是独立系统。

CORE PRINCIPLE

架构边界应由数据权限划定，不由 UI 复杂度或部署便利性划定。 当你看到 4 个"东西"时，先问它们是不是 2 个东西的 2 种渲染。

Section 02

目标架构：2 个应用，1 个 Monorepo

lingque-fusion/ ├── packages/ # 共享代码 │ ├── ui/ # 设计组件（按钮、卡片、布局） │ ├── api-client/ # 类型安全 API Client（3 个后端子系统） │ ├── auth/ # JWT 验证 + RBAC 中间件 │ └── types/ # 共享领域类型（Agent, Task, Client） │ ├── apps/ │ ├── office/ # 表面 A：客户侧（PC + PWA） │ │ └── src/app/ # 10 页已建成 │ │ ├── page.tsx # 今日概览 │ │ ├── ai-team/ # AI 团队 + 专家工作站 │ │ ├── tax/ clients/ ... # 报税 / 客户 / 合规 / 审计 / 报告 │ │ ├── skills/ # 技能商店 (OpenClaw) │ │ └── settings/ # 系统设置 │ │ │ └── internal/ # 表面 B：平台内部 │ └── src/app/ │ ├── (ops)/ # 运营路由组：客户健康 / Agent 指标 / 账单 │ ├── (expert)/ # 专家路由组：KG Explorer / 推理链 / 规则调试 │ └── (shared)/ # 共享路由组：登录 / 设置 │ └── [后端子系统不变]

为什么是 2 而不是 4

原则应用
架构边界 = 数据权限边界客户看自己的数据；内部看全部数据。这才是真正的边界
Expert + Ops 是同一类用户的不同深度都是"运营平台的团队" -- 按 Tab 分区，不按 App 分离
Mobile 是渲染变体而非新关注点同样的信息需求（委托工作），不同的密度。PWA + 微信优先
1 人团队只能维护 2 个应用界面通货膨胀（Interface Inflation）是产品死亡的真实原因
Section 03

表面定义与设计语言

表面 A：Office（客户工作台）

维度定义
用户财税公司老板、CFO、财务总监
隐喻走进虚拟办公室，7 个 AI 员工在工作
信息密度中（聊天 + 看板 + 审批）
交互模式委托式（"帮我做 XX"）
数据权限仅自己公司的 Agent、任务、工作流；仅命名 KG 查询
设计语言Digital Atelier -- Heritage Blue + Prestige Gold + Cream，充足留白，低密度

移动策略（Phase 6+）：PWA manifest + 微信服务号模板消息推送 + H5 审批页。 移动端只做 3 件事：审批 / 异常感知 / 快速对话。Native App 仅在验证实际使用模式后考虑。

表面 B：Internal（内部控制台）

路由组角色要求功能
(ops)admin多租户客户健康矩阵、Agent 性能看板、计费管理、系统告警、技能策展
(expert)expert 或 adminKG Explorer (D3.js)、推理链查看器、合规规则调试器、数据质量审计
(shared)任意内部角色登录、个人设置

设计语言：Bloomberg Ops（运营）+ Terminal Academia（专家）。 深色侧边栏、高密度表格、状态色放大、KG Explorer 使用力导向图 + 等宽标签。

Section 05

认证与权限：单 JWT，三角色

interface JWTPayload { sub: string // 用户 ID role: 'customer' | 'expert' | 'admin' // 单一角色 org_id: string // 客户公司 ID 或 'platform' }
角色apps/officeinternal (ops)internal (expert)KG T1KG T2KG T3
customer完全访问403403YesNoNo
expert只读（调试）403完全访问YesYesNo
admin完全访问完全访问完全访问YesYesYes

前端 middleware 负责路由级访问控制。后端负责数据级访问控制（基于 org_id 的行级安全）。 不需要 OAuth Provider 或 Authorization Server -- 3 个角色不值得那个复杂度。

Section 06

构建优先级

P0 -- 立即

Office 迭代

10 页已建成。下一步：添加 accountability 层（谁做了什么决策、出错通知机制），让老板信任 AI 的工作。

P1 -- 下一冲刺

Internal (Ops)

运营可见性 = 扩张前提。先建 3 页：客户健康矩阵、Agent 性能看板、系统告警。没有它只能服务 5 个客户。

P2 -- P1 之后

Internal (Expert)

迁移遗留 D3.js KG Explorer 到 Internal App。新增推理链查看器。连接 KG Semantic Layer T2 端点。

P3 -- Phase 6+

移动 PWA + 微信

先用真实客户验证移动使用模式 3 个月。PWA manifest + 微信服务号 + H5 审批页。Native App 仅在验证后考虑。

EXPLICITLY NOT BUILDING
  • 独立 Expert Workbench 应用（合并进 Internal App 的 Tab）
  • 独立 Mobile App（PWA + 微信服务号代替）
  • API Gateway（直连 3 个后端子系统，避免编织失败域）
  • OAuth Authorization Server（单 JWT 足够）

Section 08

蜂群顾问贡献

Peter Drucker
产品战略 / 客户定义 / 优先级
"管理的前提是可见性。S3（运营后台）最紧迫 -- 看不到客户健康状态，谈什么扩张？Mobile 推迟 6 个月，用企业微信 webhook 代替。"
Steve Jobs
用户体验 / 设计系统 / 品牌
"一套 Tokens，四套 Visual Language。品牌一致性不在颜色和圆角，在术语和状态色。Mobile 只做 3 件事：审批、异常、快聊。其他全砍。"
Rich Hickey
技术架构 / 简单性 / 状态管理
"4 表面 → 3 关注点 + 1 渲染变体 → 2 个应用。不要加 API Gateway -- 那是把 3 个独立失败域编织在一起。前端不持有任何 source-of-truth 状态。"
Kenya Hara
极简审计 / 存在性检验
"Expert Workbench 没有不可替代的功能 -- 全部属于 Ops 后台的调试 Tab。一个人的团队只负担得起一张客户的脸、一张自己的脸。空白本身就是潜力。"
