4. Critical Issues Identified

P0: Graph Connectivity Crisis — 91.7% Orphan Rate

54,275 out of 59,170 LawOrRegulation nodes have zero edges. Total edges 45,864 for 100,239 nodes = 0.46 edges/node. A healthy knowledge graph should have 3-5 edges/node. The "graph" in "knowledge graph" is essentially non-functional for most nodes.

Impact: Graph RAG (traversal-based retrieval) cannot leverage these nodes. Only vector search works, which defeats the purpose of a graph database.

Root Cause: Injection scripts use CREATE but don't create relationship edges. Cross-product matrix generators and AI synthesis pipeline skip edge creation.

P0: Low-Quality Template Nodes — ~13,000 Single-Sentence Placeholders

city_tax_guide (3,800) + city_industry_guide (2,000) + regional_tax_guide (1,900) + risk_monitoring (1,900) + regional_industry (1,350) + lifecycle_guidance (810) + other = ~12,760 nodes with avg content of 56-64 characters. These are essentially: "[City] [Tax]办理指南: 登录当地电子税务局，按照税务机关要求完成申报缴纳。" — identical template with city name swapped.

Impact: Zero information value. Pollutes vector index with noise. Inflates node count without substance.

P1: Duplicate Titles — 2,867 Nodes

4.8% of LR nodes share identical titles. This may indicate duplicate data from overlapping crawl sources or re-injection runs.

P1: LanceDB Vector Index Severely Outdated

13,445 vectors for 100,239 nodes — only 13.4% indexed. Vector search (the primary retrieval method) misses 86.6% of the knowledge base.

P2: Missing Entity Types — StandardCase & BusinessScenario

Original 3-layer architecture defined StandardCase and BusinessScenario as L2 operational entities. These tables no longer exist in the database — likely dropped during schema migration. L2 operational knowledge is stored as LawOrRegulation nodes instead.

7. M2 (500K) Pre-Requisites Before Scaling

  • P0
    Edge Enrichment Pipeline — Before adding ANY new nodes, build a batch edge-creation pipeline. Every LawOrRegulation must connect to at least TaxType, FTIndustry, and parent regulation. Target: reduce orphan rate from 91.7% to <20%. This is the #1 prerequisite for M2.
  • P0
    Purge Template Placeholders — Remove or enrich ~12,760 single-sentence template nodes. Options: (a) delete and regenerate with AI synthesis at 200+ chars, or (b) enrich with Gemini to add city-specific tax bureau contact, local rates, and special policies.
  • P0
    Full LanceDB Reindex — Current 13,445 / 100,239 (13.4%) coverage is unacceptable. Reindex all node types with Gemini Embedding 2 Preview (768d Matryoshka for 500K scale).
  • P1
    Deduplicate — Remove 2,867 duplicate-title nodes. Implement dedup check in injection pipeline.
  • P1
    Quality Gate in Pipeline — Add min content length check (>100 chars for LR, >50 chars for QA) to all injection scripts. Reject empty/trivial nodes at injection time.
  • P2
    Precision@5 Evaluation — Build 100 test queries and measure retrieval precision. Current state likely ~60-70% due to orphan nodes and sparse vector coverage. Target for M2: P@5 >= 85%.

8. Verdict

CONDITIONAL PASS — Do Not Proceed to M2 Without Fixes

M1 numerical target (100K nodes) is achieved. However, quality baseline is insufficient for production deployment. The knowledge graph has 100K nodes but functions more like a document store due to 91.7% orphan rate. ~12.7% of nodes are low-value template filler. Vector index covers only 13.4%.

Recommended path: Spend 1-2 sprints on P0 fixes (edge enrichment + template purge + reindex) before scaling to M2 (500K). Quality-first approach will compound — better edges and vectors at 100K yield exponentially better retrieval at 500K.
