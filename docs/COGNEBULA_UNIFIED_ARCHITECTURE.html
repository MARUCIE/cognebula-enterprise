CogNebula 统一平台架构

Ingest → Curate → Serve 三阶段管线设计

2026-03-20 3-Advisor Council Reviewed Hickey / Meadows / Musk
Section 0

Executive Summary

BEFORE

三座孤岛

3 个独立系统（KG Explorer + Expert Workbench + OpenClaw），各自拥有独立数据库、API、UI，零反馈回路。数据在三个系统间手动搬运，无法形成知识飞轮。

AFTER

统一管线

统一数据管线（Ingest → Curate → Serve），单一 API 进程、共享 v2.2 本体、信息反馈回路。事实从采集到发布全程可追溯。

KEY INSIGHT

价值管线

“Facts are values flowing through a pipeline, not three systems mutating the same places.”

— Rich Hickey

Section 1

现状 — 三个独立系统

System A: KG Explorer

  • 端口: 8400 (kg-node)
  • 职能: READ-ONLY 图谱探索
  • 技术: FastAPI + vis.js + KuzuDB + LanceDB
  • 数据: 407K 节点, 858K 边, 密度 2.1
  • 用户: 会计师、审计师、注册会计师考生

System B: Expert Workbench

  • 端口: 8900 (本地/Docker)
  • 职能: WRITE — 专家审核 + AI 三元组生成
  • 技术: FastAPI + React + Cytoscape.js + SQLite + KuzuDB (KA_ 命名空间)
  • 数据: 80 条已审核三元组, 23 个端点
  • 用户: 领域专家

System C: OpenClaw

  • 位置: VPS (100.106.223.39)
  • 职能: 自动化采集 + 入库管线
  • 技术: 76 个定时任务, Gemini, Telegram 机器人
  • 数据: 每日从 14 个源采集法规数据
  • 用户: 无（全自动化）

Section 5

数据流 — 事实生命周期

Stage 1: INGEST status: raw
  • 来源: OpenClaw 爬虫、专家 PDF 上传、手动场景输入
  • 入口: POST /api/v1/ingest + 质量门禁
  • 存储: SQLite staging 表（含溯源: source_url, crawl_date）
Stage 2: CURATE pending → approved/rejected
  • AI: Gemini 从原始文本生成本体 + 三元组
  • 人工: 专家通过 /curate UI 审核（通过/拒绝/编辑）
  • 存储: SQLite curation 表（含审计轨迹）
  • 自动审批: 置信度 ≥ 0.95 跳过人工审核
Stage 3: SERVE status: published
  • 已审批三元组 → KuzuDB（v2.2 本体）
  • 已审批文本 → LanceDB（向量嵌入）
  • 通过 /api/v1/query, search, graph, chat 查询
Section 6

信息反馈回路 (Meadows R1 飞轮)

1
专家审批三元组
2
按数据源计算审批通过率
3
高质量数据源获得更高的采集优先级
4
更优质的原始数据进入管线
5
AI 生成更高质量的三元组
6
专家审批更快（更少编辑）
7
知识图谱中积累更多已审批知识
8
终端用户获得更好的搜索 / RAG 体验
R1
更多使用 → 更多专家参与 → 回到步骤 1（增强回路）
“这条反馈回路是整个系统中最重要的单一干预点。没有它，三个系统将永远是三个孤立的数据死水。”
— Donella Meadows, Thinking in Systems
Section 7

KuzuDB 策略

维度决策
实例单实例部署在 kg-node (100.75.77.112), v2.2 本体 (15 节点表 + 20 边表)
KA_ 前缀已废弃 — 三元组直接写入 v2.2 表，不再使用 Know-Arc KA_ 命名空间隔离
置信度专家审批通过的三元组携带 confidence 分数，作为边/节点属性写入
缓冲区自动调整大小（不固定 1GB），由 KuzuDB 内部管理
备份每日 rsync 至 Mac + VPS 双副本

Section 10

风险矩阵

风险严重性缓解措施
KuzuDB 合并写入导致 OOM HIGH 分离写入窗口、自动缓冲区、深度限制
前端重写范围蔓延 MEDIUM Phase 1 Tab Shell 作为回退方案，增量迁移
OpenClaw 耦合 LOW 仅通过 HTTP webhook 集成，无共享进程/数据库
专家采用率 MEDIUM 先从 1 位专家开始 + 高置信度自动审批
