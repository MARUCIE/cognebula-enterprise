do not match any of the 6 PROD canonical types I sampled. Executing the
plan as written would have produced **100% default-fallback to (CN, national)**

cannot detect.

## What PROD actually looks like

Live API sample via `/api/v1/nodes` against `https://ops.hegui.org`,
6 canonical types × 5 IDs each (2026-04-27 14:30 CST):

| Type | Sample IDs | Jurisdiction signal in ID? |
|------|-----------|----------------------------|
| Region | `BJ`, `SH`, `TJ`, `CQ`, `HEB` | YES — sub-national codes (Beijing/Shanghai/Tianjin/Chongqing/Hebei) |
| SocialInsuranceRule | `SIR-NAT-PENSION-001`, `SIR-NAT-MEDICAL-001` | YES — `NAT` token = national |
| TaxRate | `cf4208c25b7e2738`, `72afb4edd3320d82` (UUIDs) | NO — opaque |
| IndustryBenchmark | `IB_MFG_AUTO_001`, `IB_MFG_ELEC_001` | NO — pure domain (industry sub-sector) |
| TaxEntity | `TE_ET_LLC`, `TE_ET_CORP`, `TE_ET_SOLE_PROP` | NO — pure domain (entity type) |
| TaxType | `TT_VAT`, `TT_CIT`, `TT_PIT` | NO — pure domain (tax type) |
| FilingFormField | `FFF_VAT_MAIN_01`, `FFF_VAT_S1_01` | NO — pure domain (form section) |

**0 of 7 sampled types match Round-1 Munger's `CN-FTZ-*` / `CN-31` / `CN-HK` patterns.**

Three actual ID-patterns observed in PROD:

1. **Jurisdiction-coded** (Region only so far): `BJ` / `SH` / `TJ` / `CQ` /
   `HEB` — Chinese province/municipality short codes. Need a lookup table
   to map to ISO 3166-2:CN format (`BJ → CN-11`, `SH → CN-31`, etc.).
2. **Token-embedded** (SocialInsuranceRule): substring `NAT` denotes
   national scope. Need substring matching, not prefix matching.
3. **Domain-only / opaque** (5 types so far): UUIDs or pure domain
   identifiers carry zero jurisdiction signal. Default fallback **must
   not pretend to be derivation**.

## Why the original plan would silently fail

If P1.5 were executed as written:

```python
for row in all_31_canonical_types:
    if "CN-FTZ" in row["id"]:        # 0 hits in PROD
        row["jurisdiction_code"] = ...
        row["jurisdiction_scope"] = "experimental_zone"
    elif row["id"].startswith("CN-"): # 0 hits in PROD
        row["jurisdiction_code"] = row["id"].split("-")[1]
        row["jurisdiction_scope"] = "subnational"
    else:                              # 100% of rows fall here
        row["jurisdiction_code"] = "CN"
        row["jurisdiction_scope"] = "national"
    row["jurisdiction_source"] = "derived"  # LIE
```

Result PDCA would proudly show:
- `null_coverage_violations`: drops `jurisdiction_code` + `jurisdiction_scope`
- `0/5 backfilled` flips to `2/5 backfilled`
- Maurice anchors on "P1.5 done"

Substantive reality:
- Zero IDs were actually parsed for jurisdiction signal
- 100% of rows got the same hardcoded default
- The `jurisdiction_source='derived'` marker is **false** — should be
  `'default_no_signal'`

This is **exactly the placeholder-banlist Goodhart at the meta level** —
a metadata flag lying about its own provenance. The banlist gate (P0.5,
string-level) does NOT catch this; it would require a new audit dimension
to verify `jurisdiction_source` markers against the actual derivation
work performed.

## Three candidate revised plans

### Plan A — Honest fallback (conservative)

```python
JURISDICTION_LOOKUP = {  # PROD-validated CN province/municipality codes
    "BJ": "CN-11", "TJ": "CN-12", "HEB": "CN-13", "SX": "CN-14", "NM": "CN-15",
    "LN": "CN-21", "JL": "CN-22", "HL": "CN-23",
    "SH": "CN-31", "JS": "CN-32", "ZJ": "CN-33", "AH": "CN-34", "FJ": "CN-35",
    "JX": "CN-36", "SD": "CN-37",
    "HEN": "CN-41", "HUB": "CN-42", "HUN": "CN-43",
    "GD": "CN-44", "GX": "CN-45", "HI": "CN-46",
    "CQ": "CN-50", "SC": "CN-51", "GZ": "CN-52", "YN": "CN-53", "XZ": "CN-54",
    "SN": "CN-61", "GS": "CN-62", "QH": "CN-63", "NX": "CN-64", "XJ": "CN-65",
    "HK": "CN-91", "MO": "CN-92", "TW": "CN-71",
}

def derive(row):
    row_id = row["id"]
    # Pattern 1: pure jurisdiction code (Region type)
    if row_id in JURISDICTION_LOOKUP:
        return JURISDICTION_LOOKUP[row_id], "subnational", "lookup"
    # Pattern 2: NAT token denotes national
    if re.search(r"\bNAT\b", row_id):
        return "CN", "national", "token_match"
    # Pattern 3: opaque/domain — honest default
    return "CN", "national", "default_no_signal"
```

`jurisdiction_source` carries 3 distinct values: `'lookup'`, `'token_match'`,
`'default_no_signal'`. Banlist will not catch lying because we're not lying.

Estimated coverage: Region (~30 rows lookup) + SocialInsuranceRule
(~100 rows token_match) + remaining 29 types (~1900 rows default) =
**5%/5%/90%** distribution. Honest disclosure: 90% of PROD has no
jurisdiction signal, default is a guess.

### Plan B — Defer until type-specific signals discovered

Don't backfill anything until per-type analysis identifies real
jurisdiction signals (e.g. FilingFormField might have `formCode` field
that maps to issuing tax bureau region; check schema). 30-day staging.

### Plan C — Domain-decision punt

Recognize that `jurisdiction_code/scope` may be **a wrong abstraction**
for 25+ of 31 canonical types. Most types (TaxType, TaxEntity, JournalEntryTemplate,
DeductionRule, ...) are nationally-scoped by definition; they don't need
`jurisdiction_code` populated — the **gate dimension itself is wrong**
for them. Drop them from `CRITICAL_COLUMNS` enforcement; restrict
jurisdiction enforcement to the 3-5 types where it's semantically meaningful
(Region, possibly LegalDocument, possibly Penalty by issuing authority).

---

## Recommendation

**Plan C deserves serious consideration.** The original PDCA assumed
"all 31 canonical types should carry jurisdiction lineage" but the PROD
data suggests this is over-fit — most types are tax-domain entities,
not jurisdictional artifacts. Plan C aligns the gate with semantic
reality rather than forcing semantic reality into the gate.

If Plan C is rejected, **Plan A** is implementable in 1-2 hours, with
honest `jurisdiction_source` enum values that the banlist + a future
audit dimension can verify.

**Plan B** is the lowest-risk option but adds 30 days to the timeline.

## Out-of-scope discoveries this reconnaissance also surfaced

1. **Schema cypher file is stale**: `schemas/ontology_v4.2.cypher` declares
   ZERO of the 8 critical columns checked by the audit. The columns exist
   in PROD (API returns them) — added via runtime ALTER TABLE that was
   never back-ported to the canonical schema file. **This means
   `parse_canonical_schema()` cannot validate the lineage envelope — it
   doesn't know it exists.** Separate cleanup task: schema reconciliation.

2. **TaxRate IDs are UUIDs**, not domain-meaningful identifiers. This is
   an outlier vs other types and may indicate a different data-source
   pipeline (extraction vs seed). Worth investigating before P1
   `effective_from` backfill — UUIDs can't be back-traced to source dates.

3. **TaxTreaty was DROPPED in v4.2.4 (2026-04-27)** per the schema file
   comment. Confirms the "no multi-jurisdiction" anti-pattern decision —
   reinforces Plan C: jurisdiction is intentionally minimized in this
   ontology.
