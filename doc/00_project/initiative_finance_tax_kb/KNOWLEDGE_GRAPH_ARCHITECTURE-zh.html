01. 系统概览

02. 三层数据架构

知识图谱按照法规--操作--合规三层组织，每层面向不同角色和问题域。三层之间通过 69 条跨层边（cross-layer edges）贯穿，形成从法律依据到风险预警的完整链路。

L1: 法规中心 (Regulation Center) -- 法律依据 核心问题: 规则是什么? 用户: 法务 / 税务分析员 TaxType (18) 增值税, 企业所得税, 消费税 ... LawOrRegulation (7,026) 财税[2026]15号, 国发[2024]8号 AccountingStandard (43) CAS 14 政府补助, CAS 16 企业合并 TaxIncentive (200+) 高新技术企业15%, 小微免税 Industry (1,500) GB/T 4754 国民经济行业分类 TaxAuthority (3,200) 国家税务总局 > 省市 > 县 AdministrativeRegion (3,200) 国家 > 31省 > 地市 > 县区 | 69 cross-layer edges | L2: 操作中心 (Operation Center) -- 实操指南 核心问题: 怎么做? 用户: 财务 / 税务会计 AccountEntry 会计分录 (journal entries) ChartOfAccount (91) CAS-based 科目表 BusinessScenario (15) 初创企业设立, 股权转让 ... FilingStep (15) 增值税申报第一步 ... TaxRateMapping 法规 -> 税率 -> 场景映射 SubAccount 科目明细 (GL 100~999) | 68 cross-layer edges | L3: 合规中心 (Compliance Center) -- 风险预警 核心问题: 有风险吗? 用户: 合规官 / CFO ComplianceRule (8) 增值税专票认证期限90天 RiskIndicator (5) 纳税系数异常, 所得率偏高 TaxCalendar 月度/季度申报截止日期 Penalty (5) 罚款金额 & 违规行为 EntityTypeProfile (6) 不同企业类型风险等级 AuditTrail 审计痕迹 & 变更日志

03. 双数据库设计

系统采用 KuzuDB + LanceDB 双引擎设计，以 Gemini Embedding v2 作为统一编码器。三者的关系是：向量搜索（LanceDB）是"入口"，提供语义匹配能力，帮助用户用自然语言找到相关节点；图遍历（KuzuDB）是"推理"，提供结构化关系遍历，在找到入口节点后沿着边发现关联知识。

KuzuDB -- 图数据库

  • 嵌入式架构，零运维成本
  • 原生 Cypher 查询语言
  • 单文件数据库，Git 友好
  • ACID 一致性，OLAP 优化
  • 47 节点表 / 76 关系表
  • 职责：结构化关系存储与图遍历推理
语义入口
⇄
图推理

LanceDB -- 向量数据库

  • 嵌入式列存储 (columnar)
  • ANN 近似最近邻搜索
  • MVCC 并发读优化
  • SQLite 友好，低开销
  • 60,566 vectors / 3072d
  • 职责：语义匹配 + 自然语言检索入口

Gemini Embedding v2 Preview -- 统一编码器

模型参数

  • 模型名: gemini-embedding-2-preview
  • 发布日期: 2026-03-10
  • 输出维度: 128~3072 (Matryoshka 可变)
  • 上下文: 8K tokens
  • 多语言: 中英日等全面支持
  • 多模态: text / image / video / audio / PDF

-- 给定"企业所得税优惠政策"，显示完整链路
MATCH p = (lr:LawOrRegulation)-[e*..3]-(ti:TaxIncentive)
WHERE lr.title CONTAINS '企业所得税'
  AND ti.type = 'reduction'
RETURN lr.regulation_number, ti.name, length(p)
ORDER BY length(p)
LIMIT 10;

-- 2026年有效政策 (temporal query)
MATCH (lr:LawOrRegulation)
WHERE lr.effective_date <= date('2026-03-18')
  AND (lr.expiry_date IS NULL OR lr.expiry_date > date('2026-03-18'))
RETURN COUNT(lr) as active_regulations;

08. 已知限制与未来演进

已知限制

KNOWN LIMITATION #1

KuzuDB 单进程锁

同一时刻只能 1 个 writer，多个爬虫/QA 生成/API 并发冲突。现有缓解方案为 flock 文件锁串行化写操作 + LanceDB 读副本 fallback + 降级模式。未来方案：PostgreSQL/ClickHouse 读副本或批处理窗口（每日 06:00-08:00）。

KNOWN LIMITATION #2

L1 孤立节点率 (~15-20%)

仅有 L1 政策，无 L2/L3 连接。根因是关键字匹配不足（TaxType/Industry 未出现在文本中）。解决路径：doc-tax 文件处理补充 L2 上下文、会计百科词条增加 L2/L3 定义、后续用 NLP 分类器替代关键字匹配。

KNOWN LIMITATION #3

VPS 数据中心 IP 被限制

6 个政府网站阻止 ColoCrossing VPS IP。方案包括 browser-automation (Midscene/Playwright)、住宅代理 (P3 阶段)、非 SPA 备选（已部分实施，如法规库 JSON API）。

KNOWN LIMITATION #4

LanceDB 仅覆盖 L1

RegulationClause 节点无向量 (47K+ 潜在节点)。原因是向量成本较高 (Gemini $0.20/1M, 3072d)。方案：按需增量嵌入 + 批处理窗口 + 维度降低 (768d for clauses)。

KNOWN LIMITATION #5

政策碎片化 (Fragmentation)

18 税种 x 31 省 x 企业类型 = 组合爆炸 (~10K+ combinations)。方案：分层加载（国家 -> Top 5 省 -> 市县 on-demand）、激励组合规则（显式 combinable flag）、HITL gate（不确定组合需人工审核）。
