1. 目标

基于 CogNebula Enterprise 构建SOTA 级中国财税知识操作系统：

  • 三层知识图谱：L1 法规中心（Regulation）+ L2 操作中心（Operation）+ L3 合规中心（Compliance）-- 107 张 KuzuDB 表，43 种节点类型，64 种边类型
  • 知识沉淀管线：7 阶段自动化 -- 发现 → 爬取（5 个活跃数据源）→ 新闻 → 处理（NER + KuzuDB）→ Obsidian（595 篇 MD）→ 向量化（LanceDB 3072d）→ 预警
  • 现有 AI-Fleet 技能生态（40+ 技能）用于内容采集 -- 零自定义爬虫代码
  • 中国专属本体：18 个税种，42 项会计准则（CAS），91 个科目表，7 个生命周期阶段，6 种企业类型
  • 混合 Graph RAG + 12 个 MCP 工具 + OpenClaw 集成
  • 每日增量更新，三级变更检测
  • 全自动化：常规操作零人工干预

设计原则

  1. 组合而非构建：编排 40+ 现有 AI-Fleet 技能，而非编写自定义爬虫
  2. 中国优先：所有数据源、模式、NER 规则、专家角色均针对中国税务体系设计
  3. 权威层级：P0 政府数据源为基准真相；媒体/社区仅作为补充

2. 问题陈述

痛点

  1. 信息碎片化：税收政策分散在 30+ 个政府网站（国家税务总局、财政部、国务院、海关总署、央行……），无统一机器可读 API
  2. 时间维度复杂性：政策有生效/失效日期、修订、属地覆盖（国家 → 省 → 市）、优惠叠加 -- 人工跟踪极易出错
  3. AI Agent 的上下文缺口：中国财税领域尚无可供 Agent 用 Cypher 查询的结构化知识图谱
  4. 合规滞后：每天都有新法规发布；18 税种 x 31 省份 = 规则组合爆炸

目标用户

用户待完成任务（JTBD）交互方式

3. 范围界定

纳入范围

  • 通过现有 AI-Fleet 技能采集内容（不编写自定义爬虫）：
    • news-aggregator-skill：扩展 finance-tax 配置 + 8 个新数据源抓取器
    • agent-reach：多平台爬取（政府门户、微信）
    • browser-automation（Midscene）：视觉驱动的 JS 页面处理（税务总局法规库）
    • multi-search-engine：17 引擎发现（百度、搜狗等）
    • deep-research：7 阶段 GoT 综合分析，应对复杂政策
    • defuddle / x-reader：从任意 URL 提取干净文章
    • wechat-article-search：微信公众号内容（镜像模式）
    • scrapling-crawler：反机器人绕过，处理受保护站点
  • KuzuDB 模式：13 种节点类型 + 15 种关系类型（中国税务本体）
  • LanceDB 向量嵌入（Gemini Embedding 2 Preview，通过 bin/embed）
  • 三级变更检测：哈希 + 行级差异 + 语义相似度
  • OpenClaw MCP 集成：5 个新的财税查询工具
  • 专家团队：2 个新角色（秦税安 -- 税务，顾财道 -- 财务）
  • 管线注册：写入 AI-Fleet pipeline-registry.json
  • 交付：Telegram 预警 + CF Pages 仪表板
  • PDF/OCR：PyMuPDF + PaddleOCR，处理政府公报
  • 编排脚本：scripts/finance-tax-crawl.sh（约 100 行，组合多技能）

不纳入范围（v1）

  • 自定义 Scrapy 爬虫（使用现有技能生态）
  • 北大法宝 pkulaw.com（机构付费墙 + 依据《刑法》第 285 条存在法律风险）
  • 裁判文书网批量爬取（可选 v2，仅选择性抓取）
  • 省级税务局爬取（与国家级高度冗余；v2 可加前 5 省）
  • 中国以外的多辖区支持（v2）
  • NER 模型微调（先用基于规则的正则）
  • 商业 SaaS 封装

法规文号模式（5 种）

模式发布机关层级正则表达式
国发[YYYY]N 号国务院0（最高）国发\[?\d{4}\]?\d+号
财税[YYYY]N 号财政部 + 税务总局联合1财税\[?\d{4}\]?\d+号
税总发[YYYY]N 号税务总局发文2税总[办]?发\[?\d{4}\]?\d+号
国家税务总局公告 YYYY 年第 N 号税务总局公告3国家税务总局公告\d{4}年第\d+号
海关通告 YYYY 年第 N 号海关总署3海关通告\d{4}年第\d+号

时间模型

  • 双时间轴：valid_time（政策有效期）+ transaction_time（知识库记录时间戳）
  • 版本链：TaxRateVersion 节点 + SUPERSEDES 边
  • 状态枚举：active（有效）| amended（已修订）| repealed（已废止）| pending（待生效）| proposed（征求意见）
  • 冲突解决：层级值越低优先级越高（国发 > 财税 > 税总）

5c. 知识沉淀管线

四阶段 Obsidian 中心管线（完整设计见 KNOWLEDGE_PIPELINE.md）：

阶段工具输入输出
1. 爬取finance-tax-crawl.sh（现有）20 个政府 / 媒体数据源data/raw/ 下的 JSON
2. 落档finance_tax_processor.py + json_to_obsidian.py原始 JSON经 NER 增强的 Obsidian MD
3. 管理Obsidian 仓库（~/Obsidian/财税知识库/）带 Wikilink + 标签的 MD 文件本地知识库
4. 发布CF Pages（ai-fleet-dashboard.pages.dev/finance-tax/）仓库导出在线可搜索仪表板

可选增强：NotebookLM（anything-to-notebooklm 技能，手动触发，用于播客 / FAQ 生成）

架构决策：采用 Obsidian 中心方案（而非 NotebookLM 中心），因为 NotebookLM OAuth 无法在无人值守 cron 中运行。详见 KNOWLEDGE_PIPELINE.md ADR-001。

8. 风险与缓解措施

风险严重程度缓解措施
《刑法》第 285 条（非法侵入计算机信息系统）高仅爬取公开数据源；遵守 robots.txt；5 秒延迟；不绕过验证码；咨询法律顾问
税务总局法规库 JS 渲染高browser-automation（Midscene 视觉驱动）；备选：chrome-bridge
政府网站改版中技能备选链：defuddle → browser-automation → chrome-bridge
无官方 API（仅 HTML）中NPC 法律库有部分 API；其余使用 multi-search-engine + 哈希校验
PDF OCR 准确率中PaddleOCR（CJK 优化）+ 人工审核队列
优惠叠加复杂性中每项优惠设 combinable 标志；不确定组合需人工介入
地方政策碎片化低4 级行政区划层级；按需懒加载省级数据
KuzuDB 10 万+ 节点低批量 CSV 加载器；必要时按辖区分区
