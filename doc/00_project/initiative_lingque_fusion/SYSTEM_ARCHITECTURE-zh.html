01

执行摘要

构建一家虚拟财税公司 -- 不是工具，不是仪表盘，而是一家完整的 AI 员工制代账事务所，代账公司可以直接"聘用"它。用户看到的是一间办公室: 部门、工位、工牌、技能树、跨部门协作工作流。幕后: CogNebula 知识图谱提供领域智能 (514K 节点, 2,901 条法规)，OpenClaw 供给技能链 (37K+ 精选技能)，灵阙引擎编排任务执行。

02

架构原则

01
公司即界面
UI 是一间办公室，不是仪表盘。每次交互都被框定为"把工作委派给同事"。
02
Agent 即员工
每个 Agent 有身份 (工牌)、工位、技能树 (来自 OpenClaw) 和工作流 (跨部门协作)。
03
OpenClaw 即 HR
技能从 OpenClaw 市场"招聘"而来，不是硬编码。新增能力 = 为现有员工招聘新技能。
04
知识图谱即公司大脑
知识图谱是公司的机构记忆 -- 所有员工查询它，没有人拥有它。
05
关注点分离
前端 (办公室 UI) / Agent 运行时 (员工) / KG 语义层 (大脑) / OpenClaw (HR/技能) 是独立子系统。
06
双界面，共享大脑
桌面端做专家级工作 (数据密集分析、KG 查询、批处理)，移动端做日常运营 (对话、审批、通知)。两端共享同一 API。
03

双界面战略 v2.1 NEW

为什么需要两个界面
代账公司用户有两种截然不同的交互模式，信息密度天差地别: 一张 200 家客户的合规网格无法"缩小"到手机屏幕，而对话式 Agent 在 27 寸显示器上也显得格格不入。两种模式不能共享一个自适应布局。
模式 场景 使用者 交互形式 信息密度
专家工作 桌面端 (办公室电脑) 会计、合规主管 深度分析、批量操作、KG 查询 高 (表格、看板、200 家网格)
日常运营 移动端 (微信小程序) 老板、客户经理 对话、审批、推送告警 低 (卡片、聊天气泡、通知)

API 协议规范

Agent 运行时返回结构化数据信封，绝不返回预渲染内容:

JSON 数据信封示例 
{
  "agent": { "id": "tax-accountant", "name": "林税安", "status": "busy" },
  "task": { "type": "vat-filing-check", "result": {...}, "confidence": 0.91 },
  "citations": [{ "source": "LawOrRegulation", "id": "FLK_xxx", "title": "..." }],
  "workflow_position": { "step": 3, "total": 5, "next_agent": "compliance-officer" }
}

桌面端将此数据渲染为带技能树 + 引用侧边栏的详细工位视图。
移动端将同一数据渲染为带置信度徽章和"查看详情"链接的聊天气泡。

移动端: 微信小程序

目标平台: 微信小程序 (非原生 App)。原因如下:

生态契合

2

零安装分发
扫码即用，无需下载安装

3

微信支付
未来计费可无缝接入微信支付

4

模板消息推送
税期提醒、审批通知等工作流告警

架构决策记录 (ADR)

v1.0 的 5 条 ADR 继续有效，v2.1 新增 4 条:

ADR-1
任务 Agent (角色身份外观模式)
Agent 以人格化身份面向用户，内部仍为任务驱动架构。
ADR-2
语义查询层 (禁止裸 Cypher)
所有 KG 查询通过命名语义查询工具访问，不暴露底层图查询语言。
ADR-3
置信度门控 (三级体系)
每个任务输出附带置信度评分，低于阈值自动触发人工复核。
ADR-4
引用链 (每个答案皆有来源追溯)
所有 Agent 输出包含完整引用链，追溯至原始法规或数据来源。
ADR-5
全新薄前端 (Next.js)
前端仅负责渲染，不包含业务逻辑。所有智能在后端 Agent 运行时。
ADR-6
Agent 身份系统 NEW
每个 Agent 拥有持久化身份 (姓名、工号、头像、座右铭)。身份存储在 PostgreSQL 中，不在 prompt 中。这赋予了"虚拟同事"体验。
ADR-7
OpenClaw 技能绑定 NEW
Agent 技能来源于 OpenClaw 蓝图，不是硬编码。新增能力 = 从市场"招聘新技能"。无需代码变更即可扩展系统能力。
ADR-8
工作流引擎 NEW
跨 Agent 工作流为声明式 DAG (JSON)，不是命令式代码。新增工作流是数据操作，不是部署操作。
ADR-9
双界面战略 NEW
桌面端 (Next.js) 做专家工作，移动端 (微信小程序) 做日常运营。两端共享同一 Agent 运行时 API。桌面优先 (Phase 0-5)，移动延伸 (Phase 6-8)。API 返回结构化 JSON 信封，绝不返回预渲染内容。
