五条结论，结论先行

本报告不再重复"内容覆盖率 51%"的原始警报。四位顾问一致认为：那个数字是测量伪像，真正的问题在 gate 定义、写入端缺失、谓词粒度和迁移顺序。

FINDING 01

「51% 内容覆盖率」是测量伪像，不是内容缺口

9 个代码图谱污染类型（ArrowFunction / Class / Method / File / Folder / Function / Interface / Section / External）和 3 个元数据类型（HSCode / Classification / MindmapNode）都混进了分母。重新定义 domain_types 后，真实缺口从 29pp 压缩到约 5–8pp。

FINDING 02

本体膨胀真实存在：83 活 vs 35 规范 vs 37 Brooks 天花板

rogue_in_prod 64 种，可分为 5 个命名桶：V1/V2 混流（5）、重复簇（4 域 × ~5 类）、SaaS 层泄漏（6）、legacy（7）、代码图谱污染（~27）。根因是写入端无验证，任何人加表都能绕过规范 spec 与 Brooks 告警。

FINDING 03

结构没问题，复杂交织（complection）在属性与谓词层

Hickey 判定：KnowledgeUnit 本体正确泛化。真正的"复杂交织之罪"在两处：legacyType 字符串列把"来源"伪装成"身份"；单一 KU_ABOUT_TAX 谓词扛了 166,466 条边，不区分"定义/阈值/流程/先例"。拆谓词 + 降 legacyType 为 SOURCED_FROM 边是关键。

FINDING 04

最高杠杆不是数据迁移，是 gate 重定义

Munger inversion 01：若粗暴把 64 rogue 合并进 KnowledgeUnit，KU 占比会从 33.9% 冲到 55%+，所有现有 coverage 指标都会 PASS——但检索精度崩塌。因此：改 gate 定义必须先于任何 DROP TABLE 或 MERGE。

FINDING 05

B 批次整改必须快照门禁化、顺序化

Meadows + Munger 共识：D1 snapshot → C gate 重定义 → P 白名单 → B3 TaxIncentive V2 合并 → B4 legacy 折叠 → B0 空表清理 → B2 V1/V2 重命名。逆序将导致 FK（外键）悬挂永久化（B0 先于快照 = 永久数据丢失）或"扫地时窗户没关"（B 批次先于白名单 = 几周内漂移重开）。

PER-EXPERT EVIDENCE · 四镜头证据

每位顾问只看到问题的一个面，四个面合起来才是问题

下列四组发现互相独立生成、交叉验证后合并为上文五条结论。原话保留，便于后续追溯。

ADVISOR · HARA · STRUCTURAL MINIMALISM

判定：RESTRUCTURE（结构性重构）

  1. 立即删除 15 个代码图谱污染类型：ArrowFunction / Class / Function / Method / File / Folder / Interface / Section / External / Document / Community / LifecycleActivity / LifecycleStage / Topic / SpecialZone——这些是索引实验残留物，0 域语义。
  2. SaaS 层泄漏隔离为独立 service graph：6 种类型属于客户成功工具链，不应在税/法规知识图里。Bi-graph（双图）分离后，税务查询不会碰到 CRM 行。
  3. 合并后删除所有 V2 后缀表：ComplianceRuleV2 / FilingFormV2 / RiskIndicatorV2 / TaxIncentiveV2。后缀命名是一种气味，规范名应归规范所有。
  4. 把 HSCode + Classification + MindmapNode 从 content_coverage 分母里剔除：这些是分类/导航节点，不是散文节点。把它们算进来会让测得的覆盖率从约 92% 掉到 51%——这 29pp 是伪像。
  5. KnowledgeUnit 最终应该按语义拆为子类型（TermDefinition / ProcedureStep / Threshold / PrecedentCitation）——这是 Wave 3 工作，不是 Wave 1。
ADVISOR · HICKEY · SIMPLE MADE EASY

判定："结构没问题，数据是交织的（data is complected）"

  1. 干掉 legacyType 节点列——这是"来源伪装成身份"。迁移到 (KU)-[:SOURCED_FROM]->(Source) 边。
  2. 拆分 KU_ABOUT_TAX（166,466 条边）成 DEFINES_TERM / SETS_THRESHOLD / PRESCRIBES_PROCEDURE / CITES_PRECEDENT。分类器可用：长度 + 词法特征 + 现有 legacyType 标签。
  3. INTERPRETS 家族 5 种谓词升为一等公民——不要合并它们，固化它们。
  4. content gap 是价值问题，不是结构问题。51% 对 KU 行来说是真的，但填充路径是 LLM（大语言模型）富化管线，不是 schema 迁移。
  5. 加新表之前冻结 83 表。未来 6 个月内，"先删后加"是底线。
ADVISOR · MEADOWS · SYSTEMS DYNAMICS

诊断：R1 强化环失衡 + B_MISSING 平衡环缺失

  • R1（强化，无上限）：临时 ETL 或 agent 作者的 schema 迁移 → 新表加入 → 规范漂移加剧 → 下次 ETL 视漂移为先例 → 更多新表。观察到的倍增周期 ~2 周。
  • B_MISSING（必须安装的平衡环）：每一次写 schema 的操作都必须通过白名单校验（对照 schemas/ontology_v4.2.cypher）。任何新类型需要规范决策 PR（合并请求），而不是一个迁移脚本的既成事实。Meadows Leverage #8：系统的规则才是杠杆，不是内容。
  • Leverage #3（目标重定义）：把 quality_score = 100 的目标换成 canonical_coverage_ratio ≥ 0.80 AND rogue_types = 0 AND over_ceiling_by ≤ 0。三个 ANDed 条件，不是单一可被 game 的指标。
  • 顺序警告：B0/B2/B3/B4 必须晚于 write-entry 白名单落地，否则漂移会在 ≤ 4 周内重开——"窗户没关在扫地"。
ADVISOR · MUNGER · INVERSION

5 个反向推导（"怎么把这件事彻底搞砸"）

  1. I1 Goodhart 指标作弊：把 64 rogues 通过批量 MERGE 重分类进 KnowledgeUnit → KU 份额 33.9% → 55%+，所有 coverage-by-title 指标都 PASS，而精度崩塌。所以：在建那条利用这个漏洞的管线之前，先把奖励作弊的指标杀掉。
  2. I2 周末推土机：B0+B2+B3+B4 一次性 --execute = 185K KU 谓词被盲 UPSERT 重连，无回滚。所以：原子化分批，每批前快照，批之间卡白名单。
  3. I3 空的规范 auto-stubs：给 16 个 missing_from_prod 类型填占位行会把 intersection 从 19 凑到 35，而事实总量没变。看起来是进展，其实是债务。所以：规范类型只在第一条真实行落地时才晋升，不提前填桩。
  4. I4 蜂群表演式综合：把 4 位顾问的 5 条建议各采纳一部分 = Frankenstein（弗兰肯斯坦式拼凑）。所以：选 2 条彻底执行。本 PDCA 采纳 Munger gate-first + Meadows whitelist，把 Hara 的 5 条中 3 条、Hickey 的 5 条中 1 条推到 Wave 2。
  5. I5 不可逆陷阱：在 D1 快照之前 DROP TABLE CPAKnowledge = 永久损失。REL（关系）表里的悬挂 FK 将无法诊断。所以：D1 快照是硬前置；任何 agent 在没有快照证据的情况下提议 B0/B2/B3/B4 都应被拒绝。
PDCA REMEDIATION PLAN · 整改路线

修 gate 在前，迁数据在后；每步都过快照

按 Munger gate-first + Meadows whitelist 的共识，顺序是硬的：D1 → C → P → B3 → B4 → B0 → B2 → H1 → H2。所有 B 批次触发词须用户明确输入，不自动执行。

三条件复合 gate 重定义

canonical_coverage_ratio ≥ 0.80 AND rogue_types_count = 0 AND over_ceiling_by ≤ 0。domain_types 分母排除代码图谱（9）、SaaS 泄漏（6）、元数据（3）。

写入白名单 guard

在 schema 迁移层安装 pre-commit hook（提交钩子）：任何 CREATE NODE TABLE 如果不在 schemas/ontology_v4.2.cypher 之内则拒绝；需要先开 PR 把新类型并入规范。

B 批次排序

D1 快照 → C0 gate 补丁 → P0 白名单 → B3 TaxIncentive V2 合并 → B4 legacy 折叠 → B0 空表清理 → B2 V1/V2 重命名 → H1 谓词拆分 → H2 legacyType 下沉为边。

HITL 触发词矩阵

D1 / C0 / P0 / B3 / B4 / B0 / B2 / H1 / H2。无触发词不执行。用户输入任一触发词即按序执行到下一个门禁。

每批后审计

POST /api/v1/audit 用新 gate 定义 → 期望：仅在 B3+B4+B0+B2 全部完成后才 verdict = PASS。对比前快照的 rogue_types 差集 = B 批次预期移除集合。

Wave 1 关闭动作

同步 schemas/ontology_v4.2.cypher（移除 36 种历史 rogue）；更新 PDCA 4 文档（PRD + SYSTEM_ARCHITECTURE + USER_EXPERIENCE_MAP + PLATFORM_OPTIMIZATION_PLAN）；滚动需求账本新增"gate 重定义 2026-04-24"；把 83 → 35±2 作为 Wave 1 终态。
