# SOP 3.2 — Front-Back Consistency & Entrypoint Audit

**Project**: cognebula-enterprise
**Date**: 2026-04-27
**Scope**: read-only audit + redline-format findings; trivial fixes inline; structural drift logged as deferred (Maurice decision required)
**Methodology**: SOP 3.2 (前后端一致性与入口检查), MVS-pattern, redline-format per `feedback_doc_audit_redline_format`

---

## TL;DR

The project has **two parallel backend modules** sharing the same production port (8400) but exposing **disjoint** route sets. Whichever module is actually running determines which half of the frontend works. This is a P0 structural drift that the audit caught — fixing it is Maurice-level decision (merge / split-formalize / deprecate-one).

Three lower-severity findings: one orphan iframe path, one Dockerfile-vs-systemd module mismatch (which is the *symptom* of the P0), and frontend pages coupled to specific backends without explicit declaration.

## Drift findings (redline format)

### Finding 1 — Two backend modules sharing port 8400 (P0 structural)

~~Single backend on port 8400.~~

**Two backend modules (`kg-api-server.py` and `src/api/kg_api.py`) both declare uvicorn `app` and both target port 8400. Dockerfile starts module A; systemd unit `configs/kg-api.service` starts module B. They cannot run simultaneously on the same host without port conflict, and neither is documented as deprecated.**

Reason: `Dockerfile:38` says `kg-api-server:app`; `configs/kg-api.service:9` says `src.api.kg_api:app`. Whichever path operator picks at deploy time silently determines which routes are reachable. The frontend depends on routes from BOTH modules (see Finding 3) — so neither single-module deployment serves the full frontend.

Evidence:
- routes unique to `src/api/kg_api.py` (21): `/api/v1/inspect/clause`, `/api/v1/inspect/clause/batch`, `/api/v1/registry/argument-roles`, `/api/v1/registry/check-jurisdiction-consistency`, `/api/v1/registry/resolve-override-chain`, `/api/v1/registry/resolve-override-chain/multiparent`, `/api/v1/registry/special-zones`, `/api/v1/argument-coverage`, `/api/v1/coverage/all`, `/api/v1/jurisdiction-coverage`, `/api/v1/provenance-coverage`, `/api/v1/graph-rag/plan`, `/api/neighbors/{node_id}`, `/api/node/{node_id}`, `/api/query`, `/api/sample`, `/api/search`, `/api/stats`, `/api/tables`, `/health`, `/inspect`, `/inspect.html`
- routes unique to `kg-api-server.py` (20): `/api/v1/admin/{alter-table, enrich-edges, execute-ddl, fix-titles, migrate-table, reset-table}`, `/api/v1/{chat, constellation, constellation/type, debug/paths, graph, health, hybrid-search, ingest, nodes, ontology-audit, quality, reasoning-chain, search, stats}`, `/kg_explorer.html`, `/kg_explorer_v2.html`
- **route overlap: ZERO**

### Finding 2 — Dockerfile vs systemd module path mismatch (P0, symptom of #1)

~~Dockerfile and systemd service start the same module.~~

**`Dockerfile:38` (`kg-api-server:app`) and `configs/kg-api.service:9` (`src.api.kg_api:app`) start different Python modules. Switching the deploy mechanism silently switches the route surface.**

Reason: this is the deployment-side manifestation of Finding 1. There is no doc declaring which path is canonical for which environment. Production VPS at contabo runs systemd (Backend B); Docker would run Backend A.

### Finding 3 — Frontend pages coupled to specific backends (P1)

~~Frontend pages call routes available on whichever backend is running.~~

**Frontend pages each target a specific backend without declaring which:**
- `src/web/inspect.html` calls `/api/v1/inspect/clause` and `/api/v1/inspect/clause/batch` → **only Backend B** (`src/api/kg_api.py`) declares these
- `src/web/kg_explorer.html` and `src/web/kg_explorer_v2.html` call `/api/v1/graph`, `/api/v1/nodes`, `/api/v1/search`, `/api/v1/stats`, `/api/v1/health` → primarily **Backend A** (`kg-api-server.py`); B has different `/api/search`, `/api/stats` (no `v1` prefix)
- `src/web/unified.html` iframes `/api/v1/ka/` (see Finding 4)

Reason: nginx in `deploy/contabo/nginx-cognebula.conf:71-72` blindly forwards `/api/v1/*` → `127.0.0.1:8400`. If the running backend doesn't declare a path, the frontend gets 404 with no helpful message. There is no client-side feature-detection (`OPTIONS` probe / capability map endpoint).

### Finding 4 — Orphan path `/api/v1/ka/` referenced by frontend (P2)

~~Frontend iframe sources resolve to declared backend routes.~~

**`src/web/unified.html:290` iframes `src="/api/v1/ka/"`. Neither Backend A nor Backend B declares this path. No nginx `location` rule maps it to a separate upstream. No README/doc explains its origin.**

Reason: probably an aspirational reference to a "knowledge augmentation" service that was either removed or never landed. Currently produces a broken iframe in production.

### Finding 5 — `bin/` directory missing despite README convention (P3 doc-only)

~~`bin/` directory holds CLI entrypoints per project convention.~~

**`bin/` does not exist; CLI lives in `scripts/*.sh` and `scripts/*.py`. README.md and HANDOFF.md don't reflect this. SYSTEM_ARCHITECTURE.md (if it exists) needs update.**

Reason: convention drift. Trivial doc-only fix candidate.

## Deferred half (logged, NOT fixed in this slice)

- **Finding 1 + 2**: structural decision required. Three options:
  - **Option A** — merge Backend A and B into one module. ~4000 LOC merge; needs route-by-route reconciliation (no overlap, so technically additive); risk of regression on prod.
  - **Option B** — formalize the split: rename modules (`kg_api_inspector.py` for B, `kg_api_explorer.py` for A), give them different ports (8400/8401), nginx `location` blocks for each prefix, document the boundary. Lower risk; frozen complexity.
  - **Option C** — deprecate one. Decide which is canonical, port missing routes, retire the other. Requires Maurice's product-direction call (which workflow ships?).
- **Finding 3**: client-side feature-detection / capability-map endpoint. Each frontend should call `/api/v1/.well-known/capabilities` or equivalent on load and render only the buttons whose backend routes are reachable. Sprint G candidate (build the probe FIRST, then add it to nginx routing, then refactor frontends).
- **Finding 4**: investigate `/api/v1/ka/`. Either implement (if it's a real planned service), remove (if dead), or document (if external).
- **Finding 5**: doc sync. Either create `bin/` and move CLI into it, or update README to point to `scripts/`. Trivial but out of audit scope.
