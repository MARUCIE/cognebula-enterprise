M1 Key Takeaways

  • Quality Gate 救了项目 — 如果直接推 500K，50 万个孤岛节点的图谱毫无价值
  • 模板填充是技术债 — 单句占位符（avg 56 chars）降低整体信噪比，向量检索污染
  • 边创建不能事后补 — 批量补边遇到 schema 约束、ID 匹配、方向等问题。必须在注入时同步创建
  • LanceDB + KuzuDB 互锁 — 向量重建 6 小时锁住 DB，期间无法做任何查询或评测

2. M2 Architecture Principles

PrincipleM1 做法M2 改进
Edge-First 先注入节点，事后补边 每个注入脚本必须同时创建节点+边。无边则拒绝注入。
Quality Gate 100K 后才做质量检查 每 50K 增量做一次 mini quality gate（P@5 + orphan check）
Content Minimum 无限制，56 char 也能入库 LawOrRegulation: >= 100 chars; QA: >= 50 chars; 其他: >= 30 chars
Incremental Index 全量重建 6 小时 增量嵌入：新节点即时嵌入，不锁 DB。只在 schema 变更时全量重建。
Dedup at Source 事后发现 2,867 重复 注入前 title hash 检查，重复则 MERGE（更新）而非 CREATE

需要新增 ~412,300 节点。按数据源质量分 3 个阶段推进。

Phase 1: Clause-Level Deep Split（+180K nodes）

P1 — 4 weeks

最高质量的增长路径。将现有法规拆分到条款/款/项级别，每个子条款生成 QA 对。全部 L1 质量。

Data SourceNodesQualityEdge Strategy
Regulation → Article/Paragraph/Item split ~120,000 L1 Authoritative CLAUSE_OF → parent regulation; LR_ABOUT_TAX → tax type
Clause-level QA generation (Gemini) ~60,000 L2 Operational DERIVED_FROM → source clause; LR_ABOUT_TAX → tax type
Why this first: 拆分现有法规是零成本（无需爬取），全部 L1 权威来源，每个子条款自然继承父法规的边关系。120K 高质量条款级节点 > 300K 模板填充。
