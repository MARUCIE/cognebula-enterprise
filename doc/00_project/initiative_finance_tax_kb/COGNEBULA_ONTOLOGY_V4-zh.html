01

设计原则 -- 三大支柱

本体设计遵循三个核心原则：最小节点集（21 ≤ 25 上限）、显式语义边（52 种精确关系）、单一命名规范（无版本后缀）。
01

最小节点集

21 种节点类型覆盖完整业财税领域。每新增一种必须回答：它能被现有节点的属性覆盖吗？如果能，不加。

案例：v4.1 新增 SocialInsuranceRule 而非合并入 ComplianceRule，因为双比例+上下限费率结构与 ComplianceRule schema 不兼容。

02

显式语义边

52 种边类型，每种只连接特定的 From-To 节点对。v3.1 最大改进：将 EXPLAINS（475K 条，占总边 55%）拆分为 6 种精确关系。v4.1 新增 12 种边覆盖社保/发票/行业基准/优惠互斥/关联方。

案例：INTERPRETS（解释法规）vs WARNS_ABOUT（风险提示）vs GUIDES_FILING（申报指南）。

03

单一命名

所有节点使用 PascalCase 基础名，无版本后缀。版本号属于本体版本（v4.1），不属于节点名。

修正：ComplianceRuleV2 → ComplianceRule，FilingFormV2 → FilingForm。

设计参考

FIBO（金融行业本体，W3C/OWL，70+ 领域）| XBRL Taxonomy（会计报告分类标准）| FinRegOnt（金融监管本体）| 中国税务大数据知识工程（工程院院刊 2023）| CPA/CTA 考试知识框架

02

四层架构 -- 21 种节点类型

节点按领域职责分为四层：L1 法规层（法律文件 + 条款 + 发布机构）、L2 业务层（税种/税率/科目/分类/申报）、L3 合规层（规则/指标/优惠/处罚/审计/税会差异/社保/发票/行业基准）、知识层（知识单元）。

L1 法规层 -- 法律法规的数字化表达

N1
LegalDocument
法律法规文件
合并 LawOrRegulation + AccountingStandard。包含法律、行政法规、部门规章、规范性文件、会计准则。level 属性表示层级（1-4）。
~94,090 节点
N2
LegalClause
法规条款
合并 RegulationClause + DocumentSection。从法律文件中提取的逐条条款，通过 documentId 关联父文件。
~155,350 节点
N15
IssuingBody
发布机构
法规的颁布机关。如财政部、国家税务总局、国务院等。通过 ISSUED_BY 边关联到 LegalDocument。
~398 节点

L2 业务层 -- 税务业务的结构化表达

N3
TaxRate
税率事实
从 TaxRule 拆分（Hickey 审查）。合并 TaxRateMapping + TaxRateDetail + TaxRateSchedule。支持累进税率（valueExpression）。
~9,140 节点
N10
TaxType
税种
中国现行 18 个税种。5 大分类：流转税（增值税/消费税/关税）、所得税（企业所得税/个人所得税）、资源税、特定目的税、财产行为税。
19 节点
N9
AccountingSubject
会计科目
合并 ChartOfAccount + AccountingEntry + AccountRuleMapping。覆盖资产/负债/权益/成本/损益五大类。
~1,026 节点
N8
Classification
分类体系
合并 HSCode + TaxClassificationCode + Industry + IndustryRiskProfile 等 8 张表。统一的多级分类编码体系。
~61,427 节点
N11
TaxEntity
纳税主体
合并 EnterpriseType + TaxpayerStatus。纳税人类型：一般纳税人、小规模纳税人、自然人。
~34 节点
N12
Region
行政区划
合并 Region + RegionalTaxPolicy。省/市/区三级行政区域 + 地方税收政策差异。
~651 节点
N13
FilingForm
申报表
合并 FilingFormV2 + FormTemplate + TaxCalendar。申报表模板 + 征期日历规则。
~242 节点
N14
BusinessActivity
经济业务
合并 BusinessActivity + TaxRiskScenario。企业经营活动分类，通过 TRIGGERS_TAX 触发税种。
~384 节点

L3 合规层 -- 合规约束与风险管控

N4 RENAMED
ComplianceRule
合规规则
v3.1 名称: ComplianceRuleV2。从 TaxRule 拆分（Hickey 审查）。企业必须遵守的财税合规条件 + 违规后果。
~104 节点
N5 RENAMED
RiskIndicator
风险指标
v3.1 名称: RiskIndicatorV2。合并 TaxCreditIndicator + TaxWarningIndicator。税务风险预警指标及触发阈值。
~548 节点
N6 RENAMED
TaxIncentive
税收优惠
v3.1 名称: TaxIncentiveV2。合并新旧两张 TaxIncentive 表。覆盖减免税额、即征即退、加计扣除等。
~218 节点
N16 NEW
Penalty
处罚规定
补缺：KG 中已有 127 个节点，被 PENALIZED_BY 边引用，但 v3.1 DDL 遗漏了表定义。违规行为对应的罚则和处罚标准。
127 节点（已有）
N17 NEW
AuditTrigger
审计触发
补缺：KG 中已有 463 个节点，被 TRIGGERED_BY 和 AUDIT_FOR_TAX 边引用。触发税务稽查的异常指标。
463 节点（已有）
N18 NEW
TaxAccountingGap
税会差异
全新建模：CPA/CTA 考试核心领域，代账公司最高频问题。链接 AccountingSubject 和 TaxType。gapType: 永久性/时间性/方法差异。
待入库 50+ 条
N19 NEW
SocialInsuranceRule
社保公积金规则
v4.1 新增：ft-social-insurance Skill 审查 P0 缺口。双比例费率（单位+个人）、基数上下限（社平工资 60%-300%）、地区差异。insuranceType: 养老/医疗/失业/工伤/生育/公积金/残保金。
待入库（各地区 x 险种）
N20 NEW
InvoiceRule
发票管理规则
v4.1 新增：ft-invoice-manager Skill 审查 P0 缺口。覆盖认证、抵扣、转出、红字、验真、数电票。ruleType: certification/deduction/transfer_out/red_letter/verification/e_invoice。
待入库 30+ 条
N21 NEW
IndustryBenchmark
行业基准数据
v4.1 新增：ft-risk-assessment + ft-financial-statement Skill 审查 P1 缺口。行业毛利率/净利率/税负率/费用率/周转率，支持数值范围查询（KnowledgeUnit 无法支持）。
待入库（行业 x 地区 x 年份）

知识层 -- 非结构化知识单元

N7
KnowledgeUnit
知识单元
合并 CPAKnowledge + FAQEntry + MindmapNode + SpreadsheetEntry + IndustryKnowledge。type 属性区分来源（FAQ / 案例解析 / 考点 / 风险提示 / 操作指南 / 思维导图）。通过 INTERPRETS 等 6 种边关联到 LegalClause。
~189,672 节点（KG 最大表）
设计决策

不加的节点：RegulationHierarchy（用 level 属性替代）、Evidence（用节点属性替代）、RevisionChain（用 SUPERSEDES/AMENDS 边替代）、TaxTreatment（合并入 TaxAccountingGap）。原则：能用属性解决的不加节点，控制总数 ≤ 25。v4.1 新增 3 个节点均通过 29-Skill 专家团 5 路蜂群交叉审查验证，确认无法用现有节点属性替代。

29-Skill 交叉审查结论

v4.1 升级基于 29 个财税专家 Skill（284 个 Playbook）的 5 路蜂群交叉审查。每路独立评估本体与 Skill 需求的供需匹配度，汇总后形成统一升级方案。

审查路线总览

路线 Agent 角色 发现数 P0 / P1 / P2 关键结论
Route 1合规 + 风控专家83 / 3 / 2社保公积金完全空白（P0）；行业预警值无数值范围查询（P1）
Route 2会计 + 税会差异专家72 / 3 / 2科目层级缺自引用（P0）；优惠叠加/互斥关系缺失（P1）

决策记录

每个设计决策记录选择理由和放弃方案，确保可追溯。
决策 选择 放弃方案 理由
新增 TaxAccountingGapYES嵌入边属性一等节点支持 RAG 检索、搜索、版本管理；CPA/CTA 考试独立领域
保留 Penalty / AuditTriggerYES合并入 ComplianceRule已有 590 个节点；生命周期独立（处罚标准 vs 合规规则）
不加 RegulationHierarchyNO5 种法规层级节点LegalDocument.level 属性已够用；KuzuDB 属性查询高效
不加 Evidence 节点NO独立溯源节点用 sourceUrl + confidence 属性替代；避免节点爆炸
统一命名（去 V2）YES保留 V2 后缀版本号属于本体版本，不属于节点名
18 类型上限YES22+ 类型最小复杂度原则；必要时可拆分
新增 SocialInsuranceRuleYES合并入 ComplianceRule社保费率结构（双比例+上下限）与税率差异大，schema 不兼容
新增 InvoiceRuleYES合并入 ComplianceRule发票管理是独立领域（ft-invoice-manager），规则量 30+，值得独立
新增 IndustryBenchmarkYESKnowledgeUnit 标记需要数值范围查询，KnowledgeUnit 无法支持
新增 RELATED_PARTY 边YESApplication layer only合并报表+转让定价的硬依赖，3/5 路审查提出
被否决的调研建议：v4.0 调研提出 11 种新节点类型，经评估仅保留 3 种（Penalty/AuditTrigger/TaxAccountingGap）。v4.1 蜂群审查进一步提出 6 种候选，经 5 路交叉验证保留 3 种（SocialInsuranceRule/InvoiceRule/IndustryBenchmark），否决 3 种（TaxCalendarRule 用 FilingForm.frequency 替代、AuditWorkpaper 属于运行时数据、TransferPricingMethod 用 KnowledgeUnit 覆盖）。
