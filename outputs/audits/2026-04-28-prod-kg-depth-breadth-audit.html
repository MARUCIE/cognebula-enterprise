执行摘要

  1. V2 表族增殖是杠杆最高的缺口（TaxIncentiveV2 / ComplianceRuleV2 / FilingFormV2 / RiskIndicatorV2 至少 5 张 V2 表与 V1 共存）。每对 V1+V2 都是 identity-vs-occurrence 在表层级的 complecting——本体在分叉，未来摄取会静默重复计数。
  2. /api/v1/admin/migrate-table line 1708 的 props[field] = str(val)[:500] 截断是过去迁移时硬冻结到数据上的损失。LegalClause 200 抽样里 24 个节点恰好命中 500 字（12% 截断 tell）就是这条代码的可见证据。不要盲拆——先做 Chesterton's fence 调查（embedding API token 限 / Kuzu 索引膨胀 / 历史 OOM / 防御编码）。
  3. KnowledgeUnit 内容在存储层是真碎片（200 抽样均值 136 字、最大 193 字、零节点 ≥ 500）。已通过 /api/v1/nodes（不截断的端点）确认。是设计意图还是上游来源限制还未定。
  4. 56 条本体漂移仍在（93 张运行时表 vs 31 张声明表）。schema 在 prod 是 place 而不是 value，写入端没有验证 gate。
  5. Source 归因断链：除 LegalClause 用 CL_* ID 外，其他表都是 hash 风格 ID 且无 source_id 外键。检索质量与可信度都依赖来源可解析。

R1 已撤回的 framing 错误

"KnowledgeFact 与 CPAKnowledge 不可达"——这两张表在 prod schema 里根本不存在。valid_tables 列表（93 张表）里没有它们，HTTP 404 是正确响应。

"LegalDocument 内容为空"——LegalDocument 按 schema 设计就是纯 metadata 节点。实际字段是 name / type / issueDate / jurisdiction_code / source_doc_id / source_paragraph / issuingBodyId / override_chain_id / supersedes_id / argument_links_to ...。内容存在 LegalClause 里——这是正确的 values-vs-places 分解，不是 bug。

这两条撤回是本审计区别于"凑齐六个 finding"的诚实声明——把 framing 错误公开，比悄悄改 R2 更有信息价值。

方法论

探针栈

  1. API 层采样：scripts/_lib/prod_kg_client.py.nodes(type, limit=200) 抽取每类型的内容长度分布
  2. schema 有效性探针：用 /api/v1/nodes?type=<X> 的 404 响应取出 valid_tables 完整清单
  3. 字段枚举：检查代表性节点的原始响应字段名
  4. 代码 grep：在 kg-api-server.py 里找 [:500] / substring / 截断逻辑
  5. 端点行为验证：/api/v1/nodes 返回 dict(row[0]) 不截断（line 1020-1040）；/api/v1/search 在显示层应用 [:500]（line 1123 / 1140 / 1154）

为什么反证不可省

R1 的蜂群里 Munger 标 HIGH-risk 的 confirmation bias："你测的是 API 表面，不是数据本身。" 不跑反证就 ship 的代价是把"重摄 369K 节点 = 3 周项目"误诊出来——而真相是：

  • 2/6 finding 是 schema 设计正确，不是 bug
  • 1/6 finding 是 5 行代码 fix（line 1708），不是 369K 节点项目
  • 3/6 finding 才是真正的结构性问题

这正是本会话刚刚晋升的记忆"Swarm verdict 必须验证目标文件"的同类错误——Rev. 1 把 .demo 当成 prod 是同一个错误模式。本审计在 R1 → R2 之间运行了反证探针，把这条规则真正用上了。

Findings（反证后留存）

F1V2 表族增殖

症状：valid_tables 至少包含 TaxIncentiveV2 / ComplianceRuleV2 / FilingFormV2 / RiskIndicatorV2。RiskIndicatorV2 存在但找不到 V1——命名一致性也已破。

诊断（Hickey）：schema 版本被当 PLACE 而不是 VALUE。新 schema → 新表 而不是同一规范化实体的新属性。Datomic 思路：fact 累积、schema 作为 fact 的属性附加。当前形状在说"schema 是 place"。

反向（Munger）：V1+V2 分离有合法场景——(a) V1 仍有无法原子迁移的下游消费者；(b) V2 schema 与 V1 在语义上分叉（不只是字段重命名）；(c) V2 是带回滚预案的实验。盲整合的本能在这三种情况下是错的。

下一轮工作：

  • 枚举所有 V2 表（当前 ≥5，全量未知）
  • 每对 V1+V2 分类为 (i) 仅字段重命名 → 整合 / (ii) 语义分叉 → 保持分离但写入文档 / (iii) 实验性 → 设 sunset 标准
  • 对接受分离的对：在 schema 文档里声明整合策略（V2 何时吸收 V1）

F2migrate-table line 1708 截断

代码：props[field] = str(val)[:500]——对每个通过此端点迁移的节点的每个字段应用。

可见损害：LegalClause 200 抽样中 24 个节点恰好命中 500 字（12% 截断 tell）。所有经此路径的迁移，数据已被截至 500 字写死在库里。

Chesterton's fence（Munger 强调）：拆之前要查清楚为什么加上的。要证伪的假设：

  • Embedding API token limits（Poe / Gemini 单 chunk 有 token 上限）
  • Kuzu STRING 列索引大小限制
  • 历史大批量迁移的 OOM 防御
  • 来自一次无关事故的防御编码

下一轮工作：git log -p 查 line 1708 出生 commit + 邻域；审 migrate-table 历史调用站点；评估超 500 字源数据的重迁可行性（依 source_doc_id 是否仍可解析）；选择：升至 4000 安全裕度 / 按字段参数化 / 移除并替换为字段级声明。

F3KnowledgeUnit 内容碎片化

存储层真相：200 节点抽样 mean 136 字、max 193 字、0 节点 ≥ 500 字。/api/v1/nodes 不截断已确认。

开放问题：是 (a) 设计意图（一事一节点）/ (b) 上游来源限制（爬虫在摄取时把源文档切成块）/ (c) 意外？

下一轮工作：定位创建 KnowledgeUnit 的摄取路径；检查 5-10 个 source_doc_id 指向已知 chinatax / chinaacc 文章的节点；对比 KU 内容 vs 原段落长度；按结论选择 (a) 文档化设计意图、(b) 与爬虫负责人对齐、或 (c) 找截断站点。

F4Schema 与 runtime 漂移

数字：93 张 prod 实表 vs 31 张声明表。漂移 = 62 张表。

诊断：摄取管线缺少 schema 验证 gate。新表可经 ad-hoc DDL 创建而无需更新 ontology 声明。schema 当文档用，没当 runtime 合约用。

Hickey 重构：schema-as-value vs schema-as-place。当前是被写入端 mutate 的 place；应该是系统持有并验证的 value。

下一轮工作：枚举 62 张未声明的实表；分类为 (i) 合法但未声明 → 加入 schema / (ii) 实验 → 标 _experimental 命名空间 / (iii) 已弃 → 标 sunset；在 /api/v1/admin/execute-ddl 与 /api/v1/admin/migrate-table 加 schema 验证 gate，新表必须有声明条目。

F5Source 归因缺失

症状：只有 LegalClause 用语义 ID（CL_*）。其他表用 hash 风格自动生成 ID 且无 source_id 字段。valid_tables 没有 Source 节点类型。

后果：KG 对大多数节点回答不了"这条事实从哪来？"。Lineage 与 recall trust 都依赖 source 可解析。一旦下游 agent 想用引用回答用户问题，没有 source_id 的节点等同于无法引用——内容在库里，但说不出出处。

下一轮工作：声明 Source 节点类型，schema 含 (id, label, url, ingest_ts, ingest_pipeline)；在 KnowledgeUnit / FAQEntry / TaxRate / TaxIncentive 等加 source_id 外键；在确定可映射处回填（例如 LegalClause CL_* 已编码 source；KU 可能已有 source_doc_id；FAQEntry 子集有 12366_qa_* 前缀）。

Complecting 地图（Hickey 轴 · R2 共识）

  1. Identity-vs-occurrence 在表层级（V2 增殖）——schema 版本与表身份被绑死
  2. 迁移机制与数据语义被绑死（line 1708）——transport 端对"应保留多少"的意见属于 schema，不属于 migrate-table

反向 checklist（Munger 轴 · 下一轮启动前必须过）

下一轮 queue 的综合目标

把 schema 版本视为附加在单一规范化实体上的 value，把迁移视为纯粹身份保持的函数。

这把 F1（V2 整合）与 F2（migrate-table 去截断）绑成同一个设计意图，而不是两条平行的清理流。F4（schema-runtime gate）成为该意图的执行机制。F3 与 F5 在其后排队。
