2 · 空壳回填反模式：Round-3 警告为何被绕过

2.1 部署内容

schemas/extensions/v4.2.0_create_missing_canonicals.cypher（97 行，全 DDL）：

// Audit (2026-04-25 08:14Z) identified 16 of 35 canonical types absent in prod;
// this file creates them as empty stubs (DDL-only, no data writes).
//
// CREATE NODE TABLE IF NOT EXISTS PolicyChange(...)
// CREATE NODE TABLE IF NOT EXISTS TaxTreaty(...)
// ... 14 more tables, all empty

16 张创建的表中，10 张今天仍为 0 行（ComplianceRule, IndustryBenchmark, InvoiceRule, PolicyChange, ResponseStrategy, RiskIndicator, SocialInsuranceRule, TaxItem, TaxLiabilityTrigger, TaxTreaty）。另外 10 张本就存在的 canonical 类型行数 ≤50（canonical "TINY" 层）。

2.2 Goodhart 循环（Meadows lens）

Round 3 处方：    gate FAIL → 修真实数据缺口 → gate PASS
实际发生：        gate FAIL → 加 DDL stub → 比率↑ → gate PASS（伪造）

度量 intersection_count / total_expected_tables 有一个自由度（DDL 单独就能加分子）。系统在 48 小时内利用了这个自由度。这是 Goodhart 经典变体：当度量变成目标，度量就不再是好的度量。

2.3 漏过原因（Munger lollapalooza）

认知偏差具体表现
Incentive-caused bias 激励偏差C1 比率是可见 KPI；"修 C1"在仪表板上比"关闭差距"更醒目
Doing-something bias 行动偏差Round 3 后 4 天无差距关闭 → 心理压力 → ship "任何东西"压住报警
Authority-without-enforcement 权威无强制警告写在报告里，没有进入部署管线。软警告 + 无后果 + 重复出现 = 被忽视

Round-3 文档的 anti-pattern (a) 是完全且正确的预测。它被忽视不是因为它错，而是因为文字警告不是强制原语（enforcement primitives）。

2.4 stub-backfill 启用的 3 个新失败模式

  1. 幻觉推理曲面（HIGH）：下游 agent 通过 MCP 查询 20 个 thin 类型，得到 0 或 50 行结果，然后要么(a) 幻觉填补、要么(b) 静默降质。可证伪：7 天内记录 MCP 对 20 个 thin 类型的查询，若 ≥5% 返回 <10 行 → 失败已发生。
  2. 审计链路洗白（HIGH）：out-of-band 部署无 git commit → provenance 已坏。任何成员被问"v4.2.0 何时 ship?"无法在 30 秒内说出 SHA。
  3. SOTA 提交诱惑加速（MEDIUM-HIGH）：35/35 intersection 被 60 天后的读者（包括团队自己）读为"canonical 完整"。anti-pattern (b)（过早 LegalBench-Tax 提交）变得 2 倍更可能，因为表面绿灯移除了最后一个可见 blocker。

advisor · Hickey — 解耦（complecting）

形式 vs 实质

诊断：INTERPRETS 两天没动是因为"拆 INTERPRETS"被当作一个大重构（39 万行 + 分类器重写 + 本体委员会）而非一系列原子可 ship 的解耦。stub-DDL 20 张表是同一反模式的本体级放大：易（熟悉的 DDL）赢，简（实质）输。

下周一可 ship 的动作

用 5 个高精度词法标记拆 INTERPRETS_DEFINES（约 14 万边，35-40% 剥离）：

MATCH (s)-[r:INTERPRETS]->(t:Term|Definition|Concept)
WHERE r.source_clause CONTAINS '是指'
   OR r.source_clause CONTAINS '所称'
   OR r.source_clause CONTAINS '本办法所称'
   OR r.source_clause CONTAINS '定义为'
SET r:INTERPRETS_DEFINES

一条 Cypher MATCH+SET 迁移搞定，不需要分类器重写。下周一前可测目标：INTERPRETS 39 万 → ~25 万，INTERPRETS_DEFINES ~14 万，定义类查询的 anchor_recall +0.05 以上。

检索/打分中被缠在一起的

anchor_recall@1 与 predicate_diversity@2 缠在 composite 里。两个 Pro 都 +0.150 是因为一跳定位被模型饱和；0.31 天花板住在二跳，因为 30% 的边都叫 INTERPRETS — 检索器在第二跳实际只有一个谓词可选，无法区分"定义"/"引用"/"修订"。把 anchor_recall@1 与 traversal_precision@2 独立打分，瓶颈就藏不住了。

"这周删一根麻花辫，胜过下个季度做一份完美本体。"
advisor · Meadows — 杠杆点 #8

Balancing Loop 抗 gaming 强度

主导循环（REINFORCING，退化）：gate FAIL → DDL stub → 比率伪造 → gate PASS → 空节点积累 → 真差距扩大 → 下轮 gate FAIL 更深

杠杆点：#8（Balancing Feedback Loop Strength）— "gate→fix"循环存在但被短路；强化它对 gaming 的抵抗力比改度量（#12）或改 schema（#10）更强大。

抗 gaming 度量（加入 tests/test_kg_gate.py）

def test_no_empty_canonical_tables():
    """Stub-backfill 探测器：任何 C1 表 <50 行都是 DDL-only ghost"""
    C1_EXPECTED = ["INTERPRETS", "KU_ABOUT_TAX", "AccountingSubject",
                   "RegulationArticle", "ComplianceRule"]
    ghosts = []
    for t in C1_EXPECTED:
        n = db.execute(f"SELECT COUNT(*) FROM {t}").fetchone()[0]
        if n < 50:
            ghosts.append(f"{t}({n})")
    assert not ghosts, f"先填数据：{ghosts}"

def test_c1_canonical_min50():
    MIN_ROWS = {"INTERPRETS": 300_000, "KU_ABOUT_TAX": 100_000,
                "AccountingSubject": 1_000, "DEFAULT": 50}
    for table, min_r in MIN_ROWS.items():
        count = db.execute(f"SELECT COUNT(*) FROM {table}").fetchone()[0]
        assert count >= min_r, f"{table}: {count} 行 < 阈值 {min_r}"

这两个 test 不能被 DDL 单独满足。它们强制真实数据流。

最缺的反馈循环：corpus → KG（黑洞）

循环当前状态影响
corpus → KG完全不透明。128 + 228 个 JSON 文件，零"文件→生成节点数"计数CRITIQUE
KG → eval部分。eval 存在（waves 27-40），composite 已计算中
eval → retrieval已诊断。pivot W6 推荐但未执行弱（下游已 block）

结构性变更：加 data/ingestion-manifest.jsonl，每次 ingest 写一行：{ts, source_file, rows_written: {table: count}, duration_s}。CI gate 读 manifest：mtime <48h 的源文件必须显示 rows_written 大于阈值。无 manifest → gate 立即 FAIL。manifest 显示 0 行 → FAIL。 关闭 stub 打开的循环。

advisor · Munger — 反向（机械化非劝诫）

STOP 规则

为什么 anti-pattern (a) 在明确警告下仍被应用：团队优化了一个可测代理（intersection 比率）而非底层属性（带真实数据的 canonical 覆盖）。Goodhart 定律 48 小时执行完毕，由"无 commit 的 out-of-band 部署"润滑 — 因为有人知道它过不了 code review。

STOP 规则（操作性）： 冻结所有新 schema、新 eval run 与架构工作，直到：(1) canonical_coverage_ratio 重新定义为 Σ(row_count > 1000) / 35；(2) 新数值发布到 gate 仪表板。该数值 <0.50 时，唯一允许的工作是向 20 个 thin 类型 ingest 数据；其他一切（推理引擎、多司法管辖、LegalBench、LLM tuning）冻结至新度量越过 0.50。
"Round-3 警告失败不是因为它错，而是因为它活在散文里。不让 build 失败的警告不改变行为 — 它们成为团队在它们预测的失败发生后引用的不在场证明文档。"

填补缺失数据可解锁的 3 个客户问题

  1. 代理记账公司老板（yiclaw B2B2C 目标客户）："本月增值税应纳税额是否触及小规模→一般纳税人 500 万认定门槛?" — 需要完整 TaxRate + FilingFormField。AccountingSubject 空 = Agent 无法做账科映射 = 自动代账 zero deliverable。
  2. 灵阙平台 Agent（内部工具调用客户）："行业分类代码 6940 对应的合规检查清单?" — 需要 BusinessActivity 204 → 1,500（GB/T 4754 完整）。没有这个，灵阙 15 个 Agent 调 KG = 查到空集 = 降级为 LLM 内部知识幻觉。
  3. 代账公司经理（质检 16-24 日窗口）："客户 A 资产负债表科目余额异常波动科目?" — 需要 AccountingSubject 完整 + INTERPRETS 谓词从"垃圾桶"转为有语义的关系。

商业裁决：RETHINK — 条件性继续，强制方向切换

6 个月填数据 + 跑 eval，零端到端客户演示。Drucker 法则：如果你说不出"哪个客户今天用了它做了什么"，这是研究项目，不是产品基建。

继续投资的唯一条件：30 天客户成果目标 = "yiclaw 可以跑完一家真实企业 1 月份的增值税申报全流程，无需人工干预"。需要 AccountingSubject 1,500（公开数据，企业会计准则）+ FilingFormField 1,400（公开数据，增值税申报表）— 两者都可批量脚本导入。

30 天目标未达成 → 冻结 KG 投资，把灵阙切换为 LLM-only 路径（Gemini Flash + 结构化提示词），用真实客户先证明平台价值，再回头决定 KG 是否继续填充。

优先序 yiclaw > 灵阙 > creatorlens。yiclaw 的代账场景最直接依赖 KG 账科数据，且客户研究已完成 — 你知道谁在付钱（代理记账公司），你知道他们的痛点（41 任务月度工作流）。
