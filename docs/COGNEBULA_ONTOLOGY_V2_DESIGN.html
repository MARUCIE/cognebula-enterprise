Section 5

设计原则

01
单库三层 > 三库独立
跨层遍历是核心价值。法规废止 (L1) 触发分录调整 (L2) 再触发合规失效 (L3) 需要三跳查询。拆库 = 断裂查询链 = 失去知识图谱的核心优势。
02
Entity-Relationship-Attribute, not Table-Driven
节点是实体，边是关系，属性是事实。不是关系型数据库的表-行-列思维。每种边都要通过 "So What?" 测试 -- 如果删除这条边，哪个业务查询会断？
03
Semantic Edge "So What?" Test
v1.0 有 86 种边类型，大部分无人使用。v2.2 精简到 20 种，每种都对应一个真实的业务查询路径。宁缺毋滥 -- 0 条边的类型 (如 PENALIZED_BY) 保留是因为它代表即将填充的合规路径。
04
Three-Lifecycle Separation
TaxRate (税率事实) 每年政策调整、ComplianceRule (合规约束) 随法规生效废止、RiskIndicator (风险信号) 随稽查实践演进。三者生命周期完全不同，混在一张表里会导致更新粒度失控。

Section 7

关键教训

LESSON 01
COPY FROM CSV > API row-by-row
KuzuDB API 逐行写入在 5K 条边后 OOM 崩溃。切换到 CSV COPY 后，100K+ 条边瞬间写入，速度差异 1000x。大规模图谱构建必须走批量文件导入，绝不能用 API 逐条插入。
LESSON 02
del conn; del db = checkpoint
KuzuDB 只有在 Python 对象被正常销毁时才会触发 WAL checkpoint。kill -9 会导致 WAL 回滚、数据丢失。每个 KuzuDB 会话必须以显式 del conn; del db 结束。
LESSON 03
Density is the KPI, not node count
204K 节点、density 0.16 = 无用的数据仓库，节点之间几乎没有连接。同样的节点、density 2.0 = 可遍历的知识图谱。边密度是图谱价值的唯一真实度量。
